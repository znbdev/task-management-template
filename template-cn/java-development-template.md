    *   需要等待 XX 服务提供 API 文档
    *   需要引入 XX 中间件（Redis/MQ/ES 等）

### 技术风险
*   **性能风险**：大数据量查询可能导致响应缓慢
    *   *应对*：提前进行 SQL 优化，考虑分页/缓存/异步处理
*   **兼容性风险**：新版本可能影响旧功能
    *   *应对*：充分回归测试，保留向后兼容接口
*   **并发风险**：高并发场景下可能出现数据不一致
    *   *应对*：使用分布式锁/乐观锁/事务隔离
*   **安全风险**：SQL 注入/XSS/越权访问
    *   *应对*：参数校验、预编译 SQL、权限拦截器

### 项目风险
*   需求变更风险：中途需求调整导致返工
    *   *应对*：提前确认需求边界，变更需走评审流程
*   进度风险：技术难点攻克时间超预期
    *   *应对*：预留 Buffer 时间，及时升级风险

## 💡 Java 开发最佳实践

### 分层架构规范
*   **Controller 层**：仅负责参数接收、校验、响应封装，不包含业务逻辑
*   **Service 层**：核心业务逻辑，事务控制，调用 Mapper 和其他 Service
*   **Mapper 层**：仅负责数据库 CRUD，复杂 SQL 写在 XML 中
*   **禁止跨层调用**：Controller → Service → Mapper，不可跳过

### 常见陷阱与规避
*   ❌ **空指针异常**：所有外部输入必须判空，使用 Optional/Objects.requireNonNull
*   ❌ **事务失效**：注意 @Transactional 只在 public 方法生效，同类调用不走代理
*   ❌ **循环查库**：批量查询代替循环单次查询，使用 IN 语句或 JOIN
*   ❌ **大事务**：事务范围尽量小，避免在事务中调用远程接口
*   ❌ **硬编码**：配置项写入 application.yml，常量统一定义

### 性能优化技巧
*   ✅ **数据库**：合理索引、避免 SELECT *、分页查询、读写分离
*   ✅ **缓存**：热点数据 Redis 缓存，注意缓存穿透/雪崩/击穿
*   ✅ **异步**：非核心流程异步处理（@Async/消息队列）
*   ✅ **连接池**：合理配置 HikariCP 参数（最大连接数/超时时间）

### 调试技巧
*   使用 IDEA Debug 模式断点调试
*   查看日志时关注 ERROR/WARN 级别
*   使用 Arthas 在线诊断生产问题（谨慎使用）
*   Postman/Apifox 进行接口测试

### 推荐工具链
*   **构建工具**：Maven / Gradle
*   **代码质量**：SonarQube / Checkstyle
*   **API 文档**：Swagger / Knife4j / YApi
*   **测试框架**：JUnit 5 + Mockito
*   **日志框架**：SLF4J + Logback
*   **JSON 处理**：Jackson / Fastjson2
*   **工具库**：Lombok / Hutool / Apache Commons
